# RetailCRM: ключи и загрузка `mock_orders.json`

## Начальная настройка на retailcrm.ru (что сделать в интерфейсе)

Ниже — логический порядок для **демо-аккаунта**. Точные подписи пунктов меню могут слегка отличаться (русский/английский интерфейс, версия), но разделы **Администрирование** и **Магазины / Ключи API** есть в типовой RetailCRM.

### 1. Регистрация и вход

1. Зайдите на [retailcrm.ru](https://www.retailcrm.ru/) и оформите **демо** (или пробный аккаунт), как в задании.
2. После активации вы получите доступ в **облако** по адресу вида  
   `https://<ваш-поддомен>.retailcrm.ru`  
   Это и есть значение для **`RETAILCRM_API_URL`** (без `/` в конце). Сохраните его — это «база» всех API-запросов.

### 2. Убедиться, что есть магазин (site)

API создания заказа часто требует параметр **`site`** — символьный код магазина.

1. Войдите в CRM под пользователем с правами **администратора** (в демо обычно так и есть).
2. Откройте **Администрирование** (шестерёнка / «Настройки» в нижнем левом углу или верхнем меню — зависит от темы).
3. Найдите раздел **«Магазины»** (иногда **«Сайты»** / **Stores**).
4. В списке магазинов найдите колонку **«Код»** (символьный код, латиница без пробелов, например `site1`, `demo-shop`, `s1`).  
   - Это значение подставьте в **`RETAILCRM_SITE`**.  
   - Если магазинов несколько — выберите тот, в который логично грузить тестовые заказы (часто основной магазин демо).

Если магазина нет — создайте новый по кнопке вроде **«Добавить магазин»**, задайте название и **код** (его потом и укажете в `.env`).

### 3. Создание ключа API (для `RETAILCRM_API_KEY`)

1. Снова **Администрирование**.
2. Раздел **«Интеграции»** → **«Ключи API»** (или **API-ключи**).
3. **«Создать ключ»** / **«Добавить»**.
4. Укажите понятное имя (например, `gbc-mock-upload`).
5. Важно: в правах доступа включите работу с **заказами** — как минимум **чтение и запись заказов** (в документации API это соответствует scope **`order_write`**; в интерфейсе это может называться галочками вроде «Заказы — просмотр / редактирование»).  
   Без записи метод `orders/create` вернёт ошибку доступа.
6. Сохраните ключ и **сразу скопируйте** строку ключа в надёжное место — потом он может отображаться частично или только при создании. Вставьте в **`.env`** как `RETAILCRM_API_KEY=...`.

**Не путайте:** логин и пароль от входа в веб-интерфейс CRM **не** подставляются в `RETAILCRM_API_KEY`. В код и в `.env` идёт только **длинная строка API-ключа**.

### 4. Проверка справочников (если загрузка упадёт с ошибкой «неизвестный код»)

В `mock_orders.json` уже зашиты символьные коды: `orderType`, `orderMethod`, `status`. В **вашем** демо справочники могут отличаться.

1. В интерфейсе откройте настройки **типов заказов**, **способов оформления**, **статусов** (обычно тоже под **Администрирование** → справочники / настройки заказа).
2. Сверьте **символьные коды** с теми, что в моке. Если не совпадают — либо поправьте данные в CRM под тест, либо задайте в `.env` переопределения:  
   `RETAILCRM_ORDER_TYPE`, `RETAILCRM_ORDER_METHOD`, `RETAILCRM_ORDER_STATUS` (см. таблицу ниже).

### 5. Заполнение `.env` и пробный запуск

1. В корне проекта: скопируйте `.env.example` в `.env`.
2. Заполните как минимум:
   - `RETAILCRM_API_URL`
   - `RETAILCRM_API_KEY`
   - `RETAILCRM_SITE` (желательно). Если пусто, скрипт попробует автоопределить site через `/api/credentials`.
3. Выполните:
   - `npm run upload:retailcrm -- --dry-run` — проверка маппинга без сети;
   - `npm run upload:retailcrm` — реальная загрузка.

Если ответ API содержит текст про **`site`** — вернитесь к шагу 2 и явно укажите код магазина. Если про **неизвестный статус/тип** — шаг 4.

## Какие переменные нужны

Добавьте в **корневой файл `.env`** (он в `.gitignore`, в GitHub не попадает). Шаблон см. [`.env.example`](../.env.example).

| Переменная | Обязательно | Как получить |
|------------|-------------|----------------|
| `RETAILCRM_API_URL` | Да (для реальной загрузки) | URL вашего облака, вид `https://ваш-поддомен.retailcrm.ru` — из адресной строки после входа в демо. **Без** завершающего `/`. |
| `RETAILCRM_API_KEY` | Да | **Администрирование** → **Интеграции** → **Ключи API** (названия могут чуть отличаться) → создать ключ с правом на чтение/запись заказов (`order_write`). Это **не** пароль от входа в CRM. |
| `RETAILCRM_SITE` | Желательно | **Символьный код магазина** (site): **Администрирование** → **Магазины** → колонка «Код» у нужного магазина. Если пусто, скрипт попробует взять первый доступный `site` из `/api/credentials`. |

### Опционально (если демо не принимает значения из JSON)

Справочники RetailCRM (тип заказа, способ оформления, статус) задаются **символьными кодами**. Если API отвечает ошибкой про неизвестный код, посмотрите актуальные коды в CRM и задайте:

| Переменная | Назначение |
|------------|------------|
| `RETAILCRM_ORDER_TYPE` | Подменить `orderType` вместо значения из `mock_orders.json`. Если не задан, скрипт попытается взять `defaultForApi` из `/api/v5/reference/order-types`. |
| `RETAILCRM_ORDER_METHOD` | Подменить `orderMethod`. |
| `RETAILCRM_ORDER_STATUS` | Подменить `status`. |
| `RETAILCRM_COUNTRY_ISO` | По умолчанию `KZ`; для РФ-демо может понадобиться `RU`. |
| `RETAILCRM_INCLUDE_CUSTOM_FIELDS` | `1` — отправлять `customFields` из мока (нужны одноимённые поля в CRM). По умолчанию кастомные поля **не** отправляются, чтобы реже ловить ошибки на пустом демо. |
| `RETAILCRM_UPLOAD_DELAY_MS` | Пауза между запросами в мс (по умолчанию `150`). |

## Команда

Из корня репозитория (после `npm ci` или `npm install`):

```bash
npm run upload:retailcrm
```

Проверка без API и без ключей (в т.ч. в CI):

```bash
npm run upload:retailcrm -- --dry-run
```

## Что делает скрипт

1. Читает [`mock_orders.json`](../mock_orders.json).
2. Маппит каждую запись в формат заказа API v5 (см. [`scripts/lib/map-mock-order-to-retailcrm.mjs`](../scripts/lib/map-mock-order-to-retailcrm.mjs)).
3. Для каждого заказа вызывает `POST /api/v5/orders/create` с телом `application/x-www-form-urlencoded`, поле `order` = JSON заказа, параметр `site` если задан.

Внешний идентификатор заказа: `gbc-mock-001` … — чтобы в логах было проще искать дубликаты при повторном запуске (повторная загрузка может создать новые заказы, если политика CRM иная).
